Restaurant meeting simulation set and first localization results

What changed:
- Added a `restaurant_meeting` conversation preset tuned for low overlap, minimal interruptions/backchannels, and stronger restaurant-style background noise.
- Added `simulation/create_restaurant_meeting_scene.py` to generate `restaurant_meeting_k{k}_scene{i:02d}.json` configs under `simulation/simulations/configs/restaurant_meeting_scene/`.
- The wrapper reuses the realistic-conversations scheduler/rendering path, enforces requested speaker count `k=2..5`, rejects scenes with too much overlap, and keeps generated segment assets under a separate asset root.

Dataset behavior:
- Speaker counts: `k=2..5`
- Turn taking is mostly single-speaker with short phrase-end overlaps only.
- Background noise is intentionally stronger than the older restaurant configs.
- If a requested high-`k` scene would be too short to realize all speakers under the low-overlap rules, the generator floors the effective duration upward.

Validation:
- `PYTHONPATH=. pytest -q tests/test_realistic_conversations.py tests/test_restaurant_meeting_scene_generator.py`
- Result: `5 passed`
- CLI smoke generation with `k=2` and `k=5` succeeded and produced valid configs plus scene assets/metadata.

Realtime pipeline notes:
- The realistic meeting scenes are materially heavier for the realtime pipeline because each speaker/noise track is rendered as sparse segment assets, so the same baseline/robust pipeline sweep is slower than on the older static scenes.
- One representative baseline pipeline run completed on `restaurant_meeting_k2_scene00`:
  - speaker-map MAE `66.80 deg`
  - mismatch rate `1.00`
  - map churn `0.23 deg`
  - fast-path RTF `8.27`
- Higher-`k` pipeline runs did not complete in time during this pass, so the current evidence is:
  - offline localization methods still work on the new scenes
  - the realtime pipeline path degrades badly and is substantially slower on the segmented meeting-style configs

Current takeaway:
- The new meeting scenes are meaningfully harder and more realistic than the older always-on static-source restaurant scenes.
- Offline localization quality is mixed by `k`, with `SSZ` strongest on `k=2` and `k=4`, and `SRP-PHAT` relatively better on the sampled `k=5` scene.
- The current realtime pipeline does not transfer cleanly to these segmented meeting scenes yet; it needs dedicated optimization/tuning if this dataset is going to be a primary validation target.

Separator/grouping + direction-assignment follow-up using oracle localization:
- Re-ran the meeting-scene separator comparison with estimated localization removed from the loop.
- The evaluator now feeds oracle per-chunk DOAs from `frame_ground_truth.json` directly into direction tracking, so the remaining metrics reflect separator quality, grouping stability, and speaker-to-direction assignment only.
- On `restaurant_meeting_k2_scene00`:
  - `Libri2Mix 8k`: mismatch `0.0448`, MAE `8.06 deg`, grouping switch rate `0.0909`, stability `0.769`, slow RTF `0.277`
  - `Libri3Mix 8k`: mismatch `0.0706`, MAE `12.71 deg`, grouping switch rate `0.3125`, stability `0.778`, slow RTF `0.267`
- On `restaurant_meeting_k3_scene00`:
  - `Libri2Mix 8k`: mismatch `0.7381`, MAE `85.14 deg`, grouping switch rate `0.75`, stability `0.571`, slow RTF `0.258`
  - `Libri3Mix 8k`: mismatch `0.2391`, MAE `26.61 deg`, grouping switch rate `0.4`, stability `0.667`, slow RTF `0.272`
- Interpretation:
  - `Libri2Mix 8k` is better on the sampled `k=2` scene.
  - `Libri3Mix 8k` is substantially better on the sampled `k=3` scene and is the stronger general separator choice for the meeting-style setup overall.
  - Decision for the current branch: standardize on `JorisCos/ConvTasNet_Libri3Mix_sepnoisy_8k` as the default separator for this workflow.

Speaker identity grouping follow-up:
- Added a new `speaker_embed_session` backend on top of the existing `mfcc_legacy` grouper.
- The new path keeps MFCC/Hungarian matching for short-term chunk stability, then adds causal rolling audio buffers and a session-local exact-cosine voiceprint registry for lookup/merge.

CPU speaker-embedding benchmark on this machine:
- `wavlm_xvector`:
  - first embed/load path: `5010.73 ms`
  - steady-state embed on a 1.0 s speech crop: mean `75.10 ms`, p95 `75.72 ms`
  - embedding dim: `512`
- `ecapa_voxceleb`:
  - did not complete a valid benchmark on this machine
  - current `speechbrain`/checkpoint path fails during model fetch/load (`custom.py` lookup failure against the upstream repo layout), so this backend is not usable yet in the active env

Grouping baseline on `restaurant_meeting` with oracle localization and `Libri3Mix 8k` separator:
- `mfcc_legacy`
  - `k=2`: switch rate `0.3125`, stability `0.7778`, assignment confidence `0.1433`, direction mismatch `0.0706`, direction MAE `12.71 deg`, grouping avg `2.14 ms/chunk`, slow RTF `0.282`
  - `k=3`: switch rate `0.4`, stability `0.6667`, assignment confidence `0.0996`, direction mismatch `0.2391`, direction MAE `26.61 deg`, grouping avg `1.89 ms/chunk`, slow RTF `0.256`
- `speaker_embed_session` with `wavlm_xvector`
  - `k=2`: switch rate `0.25`, stability `0.6111`, assignment confidence `0.1439`, direction mismatch `0.4842`, direction MAE `83.35 deg`, grouping avg `147.54 ms/chunk`, slow RTF `0.981`
  - `k=3`: switch rate `0.2667`, stability `0.6667`, assignment confidence `0.0952`, direction mismatch `0.0`, direction MAE `1.62 deg`, grouping avg `117.99 ms/chunk`, slow RTF `0.826`

Current takeaway on grouping:
- `wavlm_xvector` is heavy but still technically within a `200 ms` slow chunk budget on this machine.
- It is not a clear replacement for the current grouper yet:
  - strong improvement on the sampled `k=3` meeting scene
  - clear regression on the sampled `k=2` meeting scene
- `mfcc_legacy` remains the safer baseline today.
- If we keep pushing learned grouping, the next work should be on the session-merge/match thresholds rather than assuming the first learned backend is globally better.
